### 主要テーマ (Top 3)

1. **バックエンドのホットパスでの不要アロケーションと async/sync Mutex の混在** — 毎パケット Vec クローンが走るうえ、`tokio` タスク内で `std::sync::Mutex` をブロッキング取得しており、高負荷時のスループット低下・デッドロックリスクがある
2. **フロントエンドの `invoke` 失敗の握り潰し + 競合する unlisten** — エラーを捨てたまま状態リセットするパターンが複数あり、バックエンドとUIの状態が静かに乖離する
3. **UI の重要操作(リセット・ウォッチ登録)の発見性と誤操作保護** — リセットボタンが保護なし、ウォッチボタンがホバー限定表示と、主要機能への導線が弱い

## 1. 正しさ・バグ・並行性

### Backend

#### ### [High] reassembler.data 全体クローン (ホットパス)
- **Where**: `src-tauri/src/capture/windivert.rs:380`
- **Symptom**: `BinaryReader::from(reassembler.data.clone())` でパケット長4バイトを読むだけのためにバッファ全体を毎回コピーしている。高頻度パケット時に数百KB規模の不要アロケーションが発生する。
- **Recommendation**: `u32::from_be_bytes(reassembler.data[..4].try_into().unwrap())` で直接読めばクローン不要。

#### ### [High] right.to_vec() による二重アロケーション
- **Where**: `src-tauri/src/capture/windivert.rs:406`
- **Symptom**: 残りデータを `right.to_vec()` で新 Vec にコピーしたあと元バッファをドロップ。毎ループ全コピーでメモリプレッシャーが上昇する。
- **Recommendation**: `reassembler.data = reassembler.data.split_off(packet_size as usize)` に変えると O(1) で先頭を切り離せる。

#### ### [High] async タスクの無制限スポーン
- **Where**: `src-tauri/src/capture/windivert.rs:408`
- **Symptom**: 1フレームごとに `tauri::async_runtime::spawn` を呼ぶため backpressure が効かず、大量パケット時にタスクが積み上がる。全タスクが `process_opcode` の Mutex 待ちで滞留する。
- **Recommendation**: `spawn` をやめて `read_packets` ループ内で直接 `.await` する。

#### ### [High] BuffTick で同じバイト列を2回デコード
- **Where**: `src-tauri/src/engine/processor.rs:231–241`
- **Symptom**: `pb::BuffSnapshot::decode` と `pb::BuffTick::decode` を両方必ず実行している。前者成功後も後者が走り、パースコスト2倍かつ型違いで無害デコードされた場合に誤データが `apply_change` に渡る可能性がある。
- **Recommendation**: オペコードで両者が区別可能なら分岐する。両方試すなら `else if` で先勝ち打ち切りにする。

#### ### [High] TcpReassembler のシーケンス番号ラップアラウンド不一致
- **Where**: `src-tauri/src/capture/windivert.rs:349–350, 376`
- **Symptom**: TCP シーケンス番号は u32 でラップするが、`pkt_seq` を `usize` に拡大後 `saturating_sub` で比較している。シーケンス番号が u32 をラップした後（0付近）で正しく判定されない。`wrapping_add` を使っている箇所と比較側が不統一。
- **Recommendation**: シーケンス番号を `u32` のまま保持し、比較・加算を `wrapping_*` で統一する。

#### ### [Medium] is_paused チェックとパケット処理の間の二重ロック
- **Where**: `src-tauri/src/capture/mod.rs:40–49`
- **Symptom**: `is_paused` のためにロック取得・解放し、その後 `process_opcode` で再取得。毎パケット最低2回ロックする冗長なラウンドトリップ。
- **Recommendation**: `is_paused` チェックを `process_opcode` 先頭に統合し、ロック取得を1回にまとめる。

#### ### [Medium] lock poisoning を隠蔽して処理続行
- **Where**: `src-tauri/src/engine/history.rs:16–18, 28–31`
- **Symptom**: `p.into_inner()` でポイズンされたロックから内部データを取り出して続行している。データ整合性を保証できない状態での処理継続を意味する。
- **Recommendation**: `Err` を返すか、ログ出力だけして `return` する。

#### ### [Medium] subnet_reassemblers の古いエントリが永続する
- **Where**: `src-tauri/src/capture/windivert.rs:149–150, 271–285`
- **Symptom**: `MAX_SUBNET_CONNECTIONS=16` の上限があるが、接続が切れても削除されない。長時間稼働時に上限まで埋まり、新しいサブネット接続が追跡されなくなる。
- **Recommendation**: FIN/RST 検知でエントリをクリーニングするか、LRU/最終アクティブ時刻で退避させる。

#### ### [Medium] BinaryReader::is_empty が cursor 位置を無視
- **Where**: `src-tauri/src/capture/binary_reader.rs:66–71`
- **Symptom**: `is_empty()` が `len() == 0`(Vec 全長) を使っており、途中まで読んだ後に呼ぶと「まだ未読データがあるのに false」にはならないが、「全部読み終わった後も false」になる。`len()` と `remaining()` が混同されやすい。
- **Recommendation**: `is_empty()` を `remaining() == 0` にする。

#### ### [Medium] name_cache: last_seen_ms が クラッシュ時にディスク反映されない
- **Where**: `src-tauri/src/engine/name_cache.rs:132`
- **Symptom**: `entry.last_seen_ms = now_ms()` は毎回更新されるが `save_locked` は `changed == true` のみ実行。クラッシュ時に `last_seen_ms` が古いままディスクに残り、30日経過後に有効なエントリが誤削除される可能性がある。
- **Recommendation**: `last_seen_ms` の更新も `changed = true` に含めるか、設計意図をコメントで明示する。

#### ### [Medium] process_player_attrs: Vec::remove(0) による O(N) シフト
- **Where**: `src-tauri/src/engine/processor.rs:692–693`
- **Symptom**: `raw_bytes.remove(0)` は Vec 全要素を前シフトする O(N) 操作。文字列は短いので実害は小さいが不必要なコスト。
- **Recommendation**: `let raw_bytes = &attr.raw_data[1..];` でスライスを渡す。

#### ### [Low] attrs.attrs.clone() の不要なクローン
- **Where**: `src-tauri/src/engine/processor.rs:280, 283`
- **Symptom**: `process_player_attrs` / `process_monster_attrs` が所有権を要求するため呼び出し元で `.clone()` が必要。パケット1件ごとにアロケーションが発生する。
- **Recommendation**: 引数を `&[pb::RawAttr]` に変えてクローンを省く。

#### ### [Low] `default_window_icon().unwrap()` で panic リスク
- **Where**: `src-tauri/src/lib.rs:206`
- **Symptom**: アイコン未設定のビルドでクラッシュする。
- **Recommendation**: `.unwrap_or_else(...)` にするか `setup_tray` で `?` で返す。

#### ### [Low] u128 → i128 → f64 のキャスト精度
- **Where**: `src-tauri/src/bridge/commands.rs:876`
- **Symptom**: `duration_ms` が u128 で最大値付近だと i128 変換で問題が起きる可能性がある(実際には数分程度の値なので実害はほぼない)。
- **Recommendation**: `duration_ms.saturating_sub(elapsed) as f64` で u128 のまま計算する。

#### ### [Low] flush() で ReadLock 保持したまま同期ファイル書き込み
- **Where**: `src-tauri/src/engine/selected_uid.rs:62–66`
- **Symptom**: `ReadLock` を保持したまま `std::fs::write` を呼ぶ。ファイル I/O 完了まで write lock を待つスレッドが滞留する。アプリ終了時のみの使用なら許容範囲だが `name_cache` も同様の構造。
- **Recommendation**: 書き込むデータをロック内でコピーし、ロック解放後に書き込む。

### Frontend ロジック

#### ### [High] unlisteners の競合状態 — アンマウントが先行するとリーク
- **Where**: `src/App.tsx:24–30`
- **Symptom**: `Promise.all(...).then((us) => { unlisteners = us; })` は非同期で解決する。コンポーネントが即アンマウントされると `onCleanup` が先に走り空の `unlisteners` を forEach するが、その後 `.then` が解決してリスナーが代入されてもクリーンアップは二度と走らない。`encounter-reset` / `3min-measure-finalized` 等がゾンビリスナーになる。
- **Recommendation**:
  ```ts
  onMount(async () => {
    const unlisteners = await Promise.all([
      wireBackendSettings(),
      wireMeasureMode(),
      wireEncounterReset(),
    ]);
    onCleanup(() => unlisteners.forEach((u) => u()));
  });
  ```

#### ### [High] resetEncounter — invoke 失敗時に UI がリセットされない
- **Where**: `src/stores/encounter.ts:168–171`
- **Symptom**: `invoke("reset_encounter")` が throw した場合 `resetPlayerWindows()` が呼ばれず、UI が古いデータを表示し続ける。エラーは呼び出し元に伝播するが呼び出し側でハンドリングされなければユーザーには何も起きていないように見える。
- **Recommendation**: `catch` でエラーをログし、UI もリセットする(または呼び出し元でエラー表示する)。

#### ### [High] clearHistory — invoke 失敗時に古い履歴が残る
- **Where**: `src/stores/encounter.ts:204–207`
- **Symptom**: `invoke("clear_history")` が失敗した場合 `fetchHistory()` が呼ばれず表示が古いままになる。
- **Recommendation**: `try/catch` を追加し、失敗時でも `fetchHistory()` で最新状態に同期する。

#### ### [Medium] start3Min / cancel3Min — invoke 失敗後も状態をリセット
- **Where**: `src/stores/measureMode.ts:28–38`
- **Symptom**: `invoke` 失敗を `.catch(e => console.error(...))` で握り潰すため invoke 失敗でも `resetPlayerWindows()` や `setMeasureModeStatus({ kind: "normal" })` が実行される。バックエンドのモード変更が失敗したのに UI 状態がリセットされ不整合が生じる。
- **Recommendation**: `catch` でエラーを再 throw するか、戻り値で成功/失敗を返して呼び出し側で判断する。

#### ### [Medium] SelfBuffSnapshot.kind が string — タイポを型で検出できない
- **Where**: `src/stores/buffs.ts:5`

#### ### [Medium] persisted.ts — JSON.parse の型検証なし
- **Where**: `src/lib/persisted.ts:6`
- **Symptom**: `JSON.parse(s) as T` はランタイム型を保証しない。localStorage に型外の値が入ると(`"abc"` → `NaN`)、`rgba(20, 20, 30, NaN)` など描画が壊れる。バージョンアップによる型変化で起きうる。
- **Recommendation**: `typeof result === typeof defaultValue` チェックを追加し、不一致なら `defaultValue` にフォールバックする。

#### ### [Medium] watchlist.ts — storage イベントリスナーが removeEventListener されない
- **Where**: `src/stores/watchlist.ts:18–25`
- **Symptom**: `window.addEventListener("storage", ...)` が `_watched` と `_excluded` それぞれで登録されるが解除手段がない。Vite HMR 時にモジュールが再評価されるとリスナーが蓄積する。
- **Recommendation**: `persisted.ts` が storage イベントを購読しない設計と不一致なのでコメントで理由を明示する。本番動作上の問題は低い。

#### ### [Low] formatNumber と formatDps が完全に同一実装
- **Where**: `src/utils.ts:1–11`

#### ### [Low] seedLocalPlayer が MAX_WATCHLIST を無視

#### ### [Low] remainingMs ?? 1 のマジックナンバー
- **Where**: `src/stores/measureMode.ts:23`
- **Symptom**: `(status.remainingMs ?? 1) <= 0` は `undefined` のとき `false` を返すことを意図しているが `1` の意味がコードから読み取れない。

#### ### [Low] 動的 import がポーリングごとに実行

## 3. UI/UX・アクセシビリティ

#### ### [High] ヘッダーコントロールボタンがアイコン文字のみで意味不明
- **Where**: `src/components/Header.tsx:314–362`
- **Symptom**: `||`(一時停止)、`R`(リセット)、`C`(コピー)、`S`(設定)という1文字ラベルは一瞬では操作の意図が判断できない。特に `R` は確認ダイアログなしで誤クリックするとデータが即消える。
- **Recommendation**: ウィンドウ幅が十分な場合は短いテキストラベルを表示する(`alwaysOnTop` と同様の手法)。少なくとも `R` は赤系背景にして破壊的操作であることを強調する。

#### ### [High] リセットボタンの誤操作保護が不均一
- **Where**: `src/components/Header.tsx:319–333`
- **Symptom**: 3minモードが `normal` のときは確認なしに即リセットされる。`active` のときだけ `window.confirm` が入る。戦闘中に誤クリックするとデータが消える。
- **Recommendation**: `header().totalDmg > 0` の場合は常に確認ダイアログを挟む。

#### ### [High] ウォッチボタンがホバーしないと存在を知れない
- **Where**: `src/components/PlayerTable.tsx:395–414`
- **Symptom**: ウォッチボタンは `hovered()` が `true` のときのみ `visibility: visible`。バフオーバーレイが主要機能であるにも関わらず、初回使用時に偶然ホバーするまでボタンの存在に気づかない。
- **Recommendation**: ウォッチ済みは常時アイコン表示、未ウォッチは薄い透過度(`opacity: 0.3`)で常時表示する。

#### ### [High] 3minモードの pending/active の視覚的区別が弱い
- **Where**: `src/components/Header.tsx:278–310`
- **Symptom**: active 時はカウントダウン数値のみ。アイコンがないため戦闘画面の視野の端では状態変化に気づきにくい。
- **Recommendation**: active 時は `▶ MM:SS` のように状態記号を先頭に付けるか、ドット形状の状態インジケーター(緑点滅=active、黄=pending)を常時表示する。

#### ### [Medium] 設定パネルが開くとプレイヤーテーブルが大幅縮小
- **Where**: `src/App.tsx:84` / `src/components/SettingsPanel.tsx:316–329`

- **Recommendation**: 設定パネルを `position: absolute` または `position: fixed` のドロワーとして実装し、テーブルとオーバーレイで共存できるようにする。

#### ### [Medium] ThreeMinResultModal の Sparkline 幅が固定 520px
- **Where**: `src/components/ThreeMinResultModal.tsx:89`
- **Symptom**: `<Sparkline width={520} ...>` がハードコードされており、ウィンドウ幅 520px 未満でモーダルからはみ出す。

#### ### [Medium] CircularBuff の中心テキストが 7px で判読困難

- **Symptom**: `display:none` ではスクリーンリーダーが状態を読み取れない可能性がある。

#### ### [Low] ドラッグ領域とボタン領域の境界が不明瞭
- **Where**: `src/components/Header.tsx:130–144`
- **Symptom**: `data-tauri-drag-region` がヘッダー全体と Stats スパン群に設定されており、視覚的に同一のエリアでドラッグとクリックが混在している。
- **Recommendation**: Stats 部分を独立した `<div data-tauri-drag-region>` に明確に分離し、ボタン近傍には付けないよう整理する。

#### ### [Low] 削除ボタン title が日本語ハードコード
- **Where**: `src/buffs/PlayerBuffRow.tsx:67`
- **Symptom**: `title="ウォッチ解除"` が直接記述されており英語ロケール切替時も日本語のまま。`PlayerTable.tsx:401` も同様。
- **Recommendation**: `title={t("unwatch")}` のように i18n 経由に統一する。

## 4. 観察された良い点

### Backend
- `process_scene_delta` でリセット境界をまたぐ delta を `break` で打ち切り、同一パケット内での混在を防ぐロジックが正確
- `should_accept` の接続フィルタリングが未確定/学習済み/確定の3状態を適切に区別しており、マルチクライアント環境での誤集計を防ぐ設計
- `SharedDivert` の `UnsafeCell` に SAFETY コメントで C API スレッド安全性の根拠を明示しており unsafe の意図が追跡可能
- `CombatStats::record_hit` の is_crit/is_lucky 組み合わせ処理が明確に分離されており集計ロジックのバグが入りにくい
- `BuffTracker` のテストが GC・残時間計算・プレイヤー分離など主要なケースを網羅している

### Frontend ロジック
- `excluded` 集合による削除管理が「削除したプレイヤーが自動追加で戻る」問題を適切に解決
- `wireBackendSettings` の `selectedUidReady` ガードにより、初期値取得前の誤った `set_selected_uid` 呼び出しを防いでいる
- `persisted.ts` の setter が `set(() => v)` 形式で関数型 T を直接渡したときの updater 誤解釈を回避している
- `isTauri()` チェックでブラウザ開発環境でも動作する設計になっている

### UI/UX
- CircularBuff の残り 1.5 秒以下でのパルスアニメーションと残り 3 秒以下の赤色切り替えは周辺視野での状態察知に優れた設計
- ToggleChip の on/off でのカラー変化は設定パネルの現在状態を一目でスキャンできる
- ヘッダーの 2px 進捗バーが 3min モード中にゲージでも残時間を把握できる二重フィードバックとして有効
- コピーテンプレートのリアルタイムプレビューはテンプレート構文を知らないユーザーへの優れたインライン支援
